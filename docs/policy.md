# 너만고 정책 결정서 (`policy.md`)

> 이 문서는 너만고 프로젝트의 도메인 정책, 권한 정책, 예외 정책을 AI AGENTS 및 개발자가 일관되게 참고하기 위한 문서다.  
> 이 문서에 명시된 정책을 필수적으로 지켜야 하고, 실제 개발 내용이나 이전 정책들과 충돌이 있는 경우 필히 확인해야 한다.
> 실제 구현 중인 기능에 대한 해당 정책이 존재하지 않는 경우 재검토 해야한다.
> 정책은 추후 변경될 수 있다는 것을 항상 유의하고, 변경이 있는 경우 이전 정책들과 충돌이 없는지 확인해야 한다.

---

## 1. 공통 원칙

### 1.1 Entity 직접 반환 금지
- Controller는 Entity를 API 응답으로 직접 반환하지 않는다.
- 모든 응답은 Response DTO를 사용한다.

### 1.2 인증 사용자 기준
- 인증이 필요한 API는 SecurityContext의 인증 정보를 기준으로 현재 사용자를 식별한다.
- 클라이언트가 전달한 `userId`를 신뢰하지 않는다.
- 본인 확인, 작성자 확인, 팀 권한 확인은 반드시 서버에서 검증한다.

### 1.3 Soft Delete 우선
- 회원, 팀, 게시글, 댓글은 원칙적으로 Soft Delete를 사용한다.
- 물리 삭제는 기본 정책으로 사용하지 않는다.
- 삭제된 데이터는 일반 사용자 조회 결과에서 제외한다.
- 회원 탈퇴로 인해 Soft Delete된 회원이 작성한 게시글, 댓글은 삭제하지 않고, 닉네임을 "익명"처리 하여 보이게 한다.
- `TeamMember`는 팀 소속 이력으로 간주하므로 물리 삭제하지 않고 상태 기반 비활성화로 처리한다.
- `TeamMember`의 비활성 상태는 탈퇴와 강퇴를 구분하지 않는다.
- `UserCategoryMembership`은 사용자-카테고리 점유 상태를 표현하는 제약 테이블이므로, 탈퇴/강퇴/팀 삭제 시 삭제한다.

### 1.4 예외 응답 원칙
- 도메인 정책 위반은 명확한 비즈니스 예외로 처리한다.
- 예외 메시지는 정책 문서에 정의된 문구를 우선 사용한다.
- 인증 실패와 인가 실패는 구분한다.

## 2. 회원/인증 정책

### 2.1 회원가입
- 이메일 기준 중복 가입을 제한한다.
- 닉네임 중복을 제한한다.
- 비밀번호는 BCrypt로 암호화하여 저장한다.

### 2.2 로그인/JWT
- 인증 방식은 JWT Access Token + Refresh Token 구조를 사용한다.
- Refresh Token은 Redis에 저장한다.
- Redis 저장 시 TTL을 설정한다.
- Access Token은 API 인증에 사용한다.
- Refresh Token은 Access Token 재발급에 사용한다.

### 2.3 로그아웃
- 로그아웃 시 Redis에 저장된 Refresh Token을 삭제한다.
- 삭제된 Refresh Token으로는 재발급할 수 없다.

### 2.4 회원 탈퇴
- 회원 row는 Soft Delete한다.
- 탈퇴 회원이 작성한 게시글/댓글은 유지한다.
- 게시판에서 탈퇴 회원의 닉네임은 `익명`으로 표시한다.
- 탈퇴 회원은 일반 로그인 및 서비스 이용이 불가능해야 한다.
- 일반 `MEMBER`가 회원 탈퇴하면 해당 사용자의 `TeamMember`는 비활성 처리한다.
- 회원 탈퇴 시 해당 사용자의 `UserCategoryMembership`은 삭제한다.
- `OWNER`인 회원은 팀원 관리 정책의 `OWNER` 탈퇴 정책을 따른다.

## 3. 팀 정책

### 3.1 팀 생성
- 로그인한 사용자만 팀을 생성할 수 있다.
- 팀 생성자는 자동으로 해당 팀의 `OWNER`가 된다.
- 팀 생성 시 `TeamMember`가 함께 생성되어야 한다.
- 팀 생성 시 생성자의 `TeamMember`는 활성 상태로 생성된다.
- 팀 생성 시 생성자의 `UserCategoryMembership`도 함께 생성된다.
- 팀은 카테고리를 가진다. 
- 각 팀에는 멤버 수에 대한 정원이 없다.
- 팀 가입 승인 처리에서 `currentMemberCount`, `maxMemberCount` 기반 로직은 사용하지 않는다.

### 3.2 팀 상태
- 팀은 최소한 다음 상태를 가진다.
    - `RECRUITING`: 모집 중
    - `CLOSED`: 모집 마감
    - `DELETED`: 삭제됨
- `RECRUITING` 상태의 팀은 가입 신청이 가능하고 팀 목록에서 조회된다.
- `CLOSED` 상태의 팀은 가입 신청이 불가능하지만 팀 목록에서 조회된다.
- `DELETED` 상태의 팀은 일반 사용자 조회에서 제외되며 가입 신청도 불가능하다.
- 팀 마감은 `CLOSED` 상태로 처리한다.
- 팀 삭제는 물리 삭제가 아니라 `DELETED` 상태로 변경하는 Soft Delete로 처리한다.
- 삭제된 팀은 일반 조회에서 제외한다.

### 3.3 팀 조회
- 팀 목록 조회는 페이징을 적용한다.
- 카테고리별 팀 조회를 지원한다.
- 삭제된 팀은 조회되지 않는다.
- 마감된 팀의 노출 여부는 화면/기능 요구사항에 따라 결정하되, 가입 신청은 불가능하다.

## 4. 팀 가입 신청 정책

### 4.1 신청 상태
- 팀 가입 신청은 다음 상태를 가진다.
    - `PENDING`: 대기
    - `APPROVED`: 승인
    - `REJECTED`: 거절
    - `CANCELED`: 신청자 본인이 취소

### 4.2 신청 가능 조건
- 로그인한 사용자만 팀 가입 신청을 할 수 있다.
- `RECRUITING` 상태의 팀에만 신청할 수 있다.
- `CLOSED` 상태의 팀에는 신청할 수 없다.
- `DELETED` 상태의 팀에는 신청할 수 없다.
- 이미 해당 팀의 팀원인 사용자는 다시 신청할 수 없다.
- 본인이 `OWNER`인 팀에도 신청할 수 없다.
- 같은 팀에 `PENDING` 신청이 이미 있으면 중복 신청할 수 없다.

### 4.3 신청 제한: 카테고리 기준
- 한 회원은 각 카테고리에서 하나의 팀에만 소속될 수 있다.
- 서로 다른 카테고리의 팀에는 각각 하나씩 소속될 수 있다.
- 한 회원이 같은 카테고리의 여러 팀에 신청하는 것은 가능하다.
- 단, 같은 카테고리에서 가장 먼저 승인된 팀에만 소속된다.
- 같은 카테고리의 한 팀에 승인되면, 같은 카테고리의 다른 `PENDING` 신청은 자동으로 `CANCELED` 처리한다.

### 4.4 재신청 정책
- `REJECTED` 상태가 된 신청은 이후 재신청할 수 있다.
- `CANCELED` 상태가 된 신청은 이후 재신청할 수 있다.
- `PENDING` 상태에서는 같은 팀에 중복 신청할 수 없다.

### 4.5 신청 취소
- 신청자 본인만 신청을 취소할 수 있다.
- `PENDING` 상태의 신청만 취소할 수 있다.
- 취소된 신청은 `CANCELED` 상태가 된다.

### 4.6 신청 관련 예외 메시지
- 이미 해당 팀에 소속된 경우: `이미 소속되었습니다`
- 본인이 OWNER인 팀에 신청한 경우: `이미 소속되었습니다`
- 같은 팀에 PENDING 신청이 있는 경우: `이미 가입 신청한 팀입니다`
- 삭제된 팀에 접근하거나 신청하려는 경우: `존재하지 않는 팀입니다`
- 마감된 팀에 신청하려는 경우: `마감된 팀입니다`

## 5. 승인/거절 및 동시성 정책

### 5.1 승인/거절 권한
- 팀 가입 신청의 승인/거절은 해당 팀의 `OWNER`만 수행할 수 있다.
- 다른 팀의 `OWNER`는 권한이 없다.
- 일반 팀원은 승인/거절 권한이 없다.
- 승인/거절 권한은 `TeamMember`의 `role`과 활성 상태를 기준으로 검증한다.

### 5.2 승인 처리
- 승인 시 다음 작업은 하나의 트랜잭션 안에서 처리한다.
    1. 신청 상태 검증
    2. 팀 상태 검증
    3. 신청자 소속 여부 검증
    4. 같은 카테고리 내 기존 소속 여부 검증
    5. `UserCategoryMembership` 생성
    6. `TeamMember` 생성
    7. 신청 상태를 `APPROVED`로 변경
    8. 같은 카테고리의 다른 `PENDING` 신청을 `CANCELED`로 변경
- `PENDING` 상태의 신청만 승인할 수 있다.
- 승인 시 `TeamMember` 생성과 `TeamApplication` 상태 변경은 반드시 하나의 `@Transactional` 경계 안에서 처리한다.
- `UserCategoryMembership` 생성은 `TeamMember` 생성 전에 수행한다.
- `UserCategoryMembership` 생성 시 DB unique constraint 위반이 발생하면 같은 카테고리 중복 소속으로 처리한다.
- `UserCategoryMembership`은 사용자-카테고리 점유 상태를 표현하기 위한 제약 테이블이다.
- `UserCategoryMembership`은 팀원 목록 조회용 테이블이 아니다.
- 같은 신청은 중복 승인될 수 없다.
- 같은 사용자와 같은 팀의 `TeamMember` 중복 생성은 애플리케이션 검증으로 먼저 막고, DB unique constraint를 최종 방어선으로 둔다.
- 같은 사용자와 같은 카테고리의 중복 소속은 `UserCategoryMembership`의 `unique(user_id, category)` 제약을 최종 방어선으로 둔다.
- `TeamMember`는 팀 내부 멤버십을 표현하고, `UserCategoryMembership`은 사용자-카테고리 단위 소속 제약을 표현한다.
- 현재 category는 문자열 기반이며, 장기적으로 enum 또는 별도 `Category` 테이블 분리를 검토한다.

### 5.3 거절 처리
- `PENDING` 상태의 신청만 거절할 수 있다.
- 거절 시 신청 상태를 `REJECTED`로 변경한다.
- `REJECTED` 신청은 이후 재신청할 수 있다.
- 거절 처리는 반드시 `@Transactional` 경계 안에서 수행한다.
- 같은 신청은 중복 거절되거나 승인 후 거절될 수 없다.

### 5.4 동시성 제어
- 같은 카테고리의 여러 팀에 동시에 승인 요청이 들어와도 한 회원은 하나의 팀에만 소속되어야 한다.
- 같은 신청의 동시 승인/거절 처리를 막기 위해 승인/거절 대상 `TeamApplication` 조회에는 `PESSIMISTIC_WRITE` 락을 적용한다.
- 같은 카테고리 중복 소속은 애플리케이션 검증과 `UserCategoryMembership` DB unique constraint로 방지하고, `TeamMember` 중복 생성은 DB unique constraint를 최종 방어선으로 둔다.
- 팀에는 멤버 정원이 없으므로 승인 동시성 설계에서 정원 기반 검증은 고려하지 않는다.

### 5.5 승인 관련 예외 메시지
- 이미 처리된 신청인 경우: `이미 처리된 신청입니다`
- 권한 없는 사용자가 승인/거절하는 경우: `팀 주장만 처리할 수 있습니다`
- 같은 카테고리의 다른 팀에 이미 소속된 경우: `이미 해당 카테고리의 팀에 소속되었습니다`

## 6. 팀원 관리 정책

### TeamMember 상태
- `ACTIVE`: 현재 활성 팀원
- `INACTIVE`: 탈퇴, 강퇴, 팀 삭제 등으로 비활성화된 팀원
- 탈퇴와 강퇴는 상태적으로 구분하지 않는다.
- 일반 조회에서는 `INACTIVE` 상태를 제외한다.

### 6.1 팀원 목록 조회
- 해당 팀의 멤버 목록을 조회할 수 있다.
- 비활성 처리된 멤버는 기본 조회에서 제외한다.

### 6.2 팀 탈퇴
- 일반 멤버는 본인이 소속된 팀에서 탈퇴할 수 있다.
- 탈퇴 시 해당 팀의 `TeamMember`는 물리 삭제하지 않고 비활성 처리한다.
- 탈퇴 시 해당 팀 카테고리에 대한 `UserCategoryMembership`은 삭제한다.
- 여러 팀에 속한 회원이 특정 팀에서 탈퇴하면 해당 팀 소속만 제거한다.
- 다른 카테고리의 팀 소속에는 영향을 주지 않는다.

### 6.3 OWNER 탈퇴
- 팀원이 OWNER 혼자인 경우 OWNER는 자유롭게 탈퇴할 수 있다.
- OWNER 혼자 탈퇴하면 팀은 자동 삭제 처리한다.
- OWNER 혼자 탈퇴하여 팀이 삭제되면 해당 팀의 `PENDING` 가입 신청은 자동으로 `REJECTED` 처리한다.
- OWNER 혼자 탈퇴하여 팀이 삭제되면 해당 OWNER의 `TeamMember`는 비활성 처리한다.
- OWNER 혼자 탈퇴하여 팀이 삭제되면 관련 `UserCategoryMembership`은 삭제한다.
- 팀원이 여러 명인 팀에서 OWNER는 바로 탈퇴할 수 없다.
- 팀원이 여러 명인 팀의 OWNER는 다음 OWNER를 위임한 뒤 탈퇴해야 한다.

### 6.4 OWNER 탈퇴 관련 예외 메시지
- 여러 명의 팀원이 있는 팀에서 OWNER가 위임 없이 탈퇴하려는 경우:  
  `현재 탈퇴하려는 팀의 주장입니다. 다음 주장을 선택하고 탈퇴하여 주십시오.`
- OWNER 혼자 탈퇴하여 삭제된 팀에 이후 접근하는 경우:  
  `존재하지 않는 팀입니다`

### 6.5 주장 권한 위임
- OWNER만 주장 권한을 위임할 수 있다.
- 위임 대상은 같은 팀의 활성 멤버여야 한다.
- 위임은 자유롭게 수행할 수 있다.
- 위임 시 기존 OWNER는 일반 멤버로 변경된다.
- 위임 시 기존 OWNER 일반 멤버 변경과 새 OWNER 승격은 하나의 트랜잭션 안에서 처리한다.
- 주장 권한 위임 이력은 별도로 저장하지 않는다.
- 한 팀에는 활성 OWNER가 반드시 한 명만 존재해야 한다.
- OWNER 단일성은 DB 제약과 트랜잭션으로 함께 보장한다.

### 6.6 팀원 강퇴
- OWNER만 다른 멤버를 강퇴할 수 있다.
- OWNER는 자기 자신을 강퇴할 수 없다.
- OWNER는 본인 팀의 멤버만 관리할 수 있다.
- 다른 팀의 OWNER 또는 멤버를 강퇴할 수 없다.
- 어느 한 팀의 OWNER인 회원이라도 다른 팀에서 일반 멤버인 경우 해당 팀의 OWNER에 의해 강퇴될 수 있다.
- 강퇴된 사용자의 `TeamMember`는 물리 삭제하지 않고 비활성 처리한다.
- 강퇴 시 해당 팀 카테고리에 대한 `UserCategoryMembership`은 삭제한다.
- 강퇴와 탈퇴는 상태적으로 구분하지 않는다.
- 강퇴된 사용자는 이후 해당 팀에 재신청할 수 있다.

### 6.7 강퇴 관련 예외 메시지
- OWNER가 자기 자신을 강퇴하려는 경우: `자기 자신을 강퇴할 수 없습니다`
- 권한 없는 사용자가 강퇴하려는 경우: `팀 주장만 처리할 수 있습니다`
- 본인 팀의 멤버가 아닌 사용자를 강퇴하려는 경우: `해당 팀의 멤버가 아닙니다`

---

## 7. 게시글/댓글 정책

### 7.1 게시글
- 로그인한 사용자는 게시글을 작성할 수 있다.
- 모든 게시글은 공개된다.
- 게시글 수정/삭제는 작성자 본인만 가능하다.
- 관리자는 운영 정책에 따라 게시글을 강제 삭제할 수 있다.
- 게시글 삭제는 Soft Delete로 처리한다.

### 7.2 댓글
- 로그인한 사용자만 댓글을 작성할 수 있다.
- 모든 게시글은 댓글을 허용한다.
- 댓글 수정/삭제는 작성자 본인만 가능하다.
- 관리자는 운영 정책에 따라 댓글을 강제 삭제할 수 있다.
- 댓글 삭제는 Soft Delete로 처리한다.

### 7.3 탈퇴 회원 표시
- 탈퇴 회원이 작성한 게시글/댓글은 유지한다.
- 탈퇴 회원의 닉네임은 게시판 화면에서 `익명`으로 표시한다.

### 7.4 게시글/댓글 예외 메시지
- 작성자가 아닌 사용자가 수정/삭제하려는 경우: `작성자만 수정 또는 삭제할 수 있습니다`
- 삭제된 게시글에 접근하는 경우: `존재하지 않는 게시글입니다`
- 삭제된 댓글에 접근하는 경우: `존재하지 않는 댓글입니다`

---

## 8. 공지사항/관리자 권한 정책

### 8.1 공지사항
- 공지사항 조회는 일반 사용자도 가능하다.
- 공지사항 등록/수정/삭제는 관리자만 가능하다.
- 공지사항 삭제는 Soft Delete로 처리한다.

### 8.2 관리자 권한
- 초기 구현에서는 `ROLE_ADMIN` 하나만 사용한다.
- 관리자 권한 세분화는 운영 단계에서 필요성이 생기면 분리한다.
- 현재는 `ROLE_ADMIN`, `ROLE_MANAGER`, `ROLE_MODERATOR`를 분리하지 않는다.

### 8.3 관리자 접근 제한
- 초기 구현에서는 고정 IP 제한을 적용하지 않는다.
- 관리자 API 접근은 사용자의 ROLE이 `ROLE_ADMIN`인 경우로 제한한다.
- IP 제한은 배포/운영 단계에서 필요하면 추가한다.

## 9. 알림 정책

### 9.1 저장형 알림
- 모든 알림은 DB에 저장한다.
- 알림은 읽음/안읽음 상태를 가진다.
- 사용자는 본인의 알림 목록을 조회할 수 있다.
- 사용자는 본인의 알림을 읽음 처리할 수 있다.

### 9.2 SSE 실시간 알림
- SSE는 실시간 전달 수단으로 사용한다.
- SSE 전달 실패가 발생해도 DB 알림은 유지한다.
- 사용자는 SSE 연결이 끊겨도 알림함에서 알림을 확인할 수 있다.

### 9.3 알림 발생 이벤트
- 팀 가입 신청이 생성되면 해당 팀 OWNER에게 알림을 생성한다.
- 팀 가입 신청이 승인되면 신청자에게 알림을 생성한다.
- 팀 가입 신청이 거절되면 신청자에게 알림을 생성한다.
- 같은 카테고리의 다른 신청이 자동 취소된 경우, 필요 시 신청자에게 알림을 생성할 수 있다.

## 10. 관리자/운영 정책

### 10.1 회원 관리
- 관리자는 회원 목록을 조회할 수 있다.
- 관리자는 회원 상태를 변경할 수 있다.
- 관리자는 정책 위반 사용자를 강제 탈퇴 처리할 수 있다.

### 10.2 게시글/댓글 관리
- 관리자는 게시글과 댓글을 강제 삭제할 수 있다.
- 관리자 삭제도 Soft Delete로 처리한다.

### 10.3 신청 관리
- 관리자는 팀 가입 신청 내역을 조회할 수 있다.
- 관리자는 운영 목적으로 신청 상태를 확인할 수 있다.

### 10.4 운영 로그
- 관리자가 데이터를 변경하는 행위는 Audit Log로 기록한다.
- 최소 기록 항목은 다음과 같다.
    - 관리자 ID
    - 대상 리소스 타입
    - 대상 리소스 ID
    - 수행 작업
    - 수행 시각

### 10.5 통계
- 관리자는 카테고리별 팀 생성 수를 조회할 수 있다.
- 관리자는 사용자 활동 통계를 조회할 수 있다.

## 11. 매치 정책

> 매치 기능은 후순위 Phase에서 구현한다. 아래 정책은 초기 기준이다.

### 11.1 매치 등록
- 팀 단위로 매치를 등록한다.
- 매치 등록은 해당 팀 OWNER 또는 정책상 허용된 팀 멤버만 가능하다.
- 구현 시점에 최종 권한을 확정한다.

### 11.2 매치 신청
- 다른 팀의 매치에 신청할 수 있다.
- 중복 신청은 방지한다.
- 매치 신청에도 동시성 제어가 필요할 수 있다.

### 11.3 매치 결과
- 경기 결과 저장은 후순위로 구현한다.
- 결과 확정 방식은 별도 정책으로 확정한다.

## 12. 구현 시 주의사항

### 12.1 트랜잭션
- 상태 변경이 여러 Entity에 걸쳐 발생하는 작업은 하나의 트랜잭션으로 처리한다.
- 승인, 탈퇴, 강퇴, 위임, 팀 삭제는 반드시 트랜잭션 경계를 명확히 한다.
- `TeamMember` 비활성 처리와 `UserCategoryMembership` 삭제는 반드시 동일 트랜잭션 안에서 처리한다.
- OWNER 위임 시 기존 OWNER 일반 멤버 변경과 새 OWNER 승격은 반드시 하나의 트랜잭션 안에서 처리한다.
- OWNER 혼자 탈퇴하여 팀이 삭제되는 경우, 팀 `DELETED` 처리, `TeamMember` 비활성 처리, `UserCategoryMembership` 삭제, `PENDING` 신청 `REJECTED` 처리는 하나의 트랜잭션 안에서 처리한다.

### 12.2 N+1 방지
- 팀 목록, 팀 상세, 팀원 목록, 게시글/댓글 조회에서 N+1이 발생하지 않도록 주의한다.
- Fetch Join, EntityGraph, DTO Projection, Querydsl을 상황에 맞게 사용한다.

### 12.3 DB 제약조건
- 애플리케이션 검증만 신뢰하지 않는다.
- 가능한 경우 Unique Constraint와 인덱스를 함께 설계한다.
- 팀 승인, 팀원 생성, 같은 카테고리 중복 소속 방지는 DB 제약과 트랜잭션 정책을 함께 고려한다.
